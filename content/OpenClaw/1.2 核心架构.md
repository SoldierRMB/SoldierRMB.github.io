## 核心架构图：消息通道 -> Gateway -> Pi Agent 运行时 -> 工具执行

![[architecture.drawio.svg]]

有几个值得关注的点：

- **通道嵌入 Gateway 进程**：通道并不是独立的服务，而是作为插件或者适配器嵌入在 Gateway 进程内部的。Gateway 在启动时会初始化所有已经配置的通道。通道与 Gateway **是进程内函数调用**，而非网络通信。
- **单一控制平面**：Gateway 是整个系统的枢纽。所有消息都嵌入在 Gateway 中，客户端通过 WebSocket 连接到 Gateway。Gateway 负责协调——路由消息、管理会话、Agent 调度、分发事件。这种设计简化了系统状态管理和一致性。
- **客户端 ≠ 消息通道**：消息通道只是面向终端用户的一个入口，而客户端是面向操作者（OpenClaw 的管理者）。客户端通过 WebSocket 连接到 Gateway 只要用于实时监控、审批危险命令、提供设备能力、以及管理会话和配置。
- **关注点分离**：通道只负责消息的收发，不参与任何业务逻辑。Agent 运行时只关注 AI 调用和工具调用，不关心消息来自哪个通道。Gateway 只负责协调。
- **工具即能力**：Agent 的能力并不是硬编码，而是通过可注册的工具来拓展。其中包括有内置工具、浏览器、Agent 协调、技能、扩展、自动化、媒体和设备。

## 关键子系统

### Gateway（网关 / 控制平面）

> **衍生解释**：控制平面（Control Plane）是一个源自网络工程和分布式系统的概念。在计算机网络中，控制平面负责决策——决定数据包应该被转发到哪里；而数据平面（Data Plane）负责执行——实际转发数据包。类似地，在微服务架构中，控制平面（如 Kubernetes API Server）负责管理和协调，而各个工作节点负责实际执行任务。
>
> 在 OpenClaw 中，Gateway 就扮演控制平面的角色。它本身不执行 AI 推理（那是 Pi Agent 运行时的工作），也不直接处理消息格式转换（那是通道适配器的工作）。它做的是**协调**：接收来自各方的请求和事件，做出路由决策，调度执行，分发结果。

Gateway 是 OpenClaw 的心脏。是一个长期运行的服务，默认在 `127.0.0.1:18789` 监听，主要职责包括：

- **WebSocket 服务器**：提供与所有客户端、节点的连接
- **HTTP 服务器**：提供与 OpenAI 兼容 AI 交互、控制面板 UI
- **通道管理**：启动和维护消息通道
- **会话管理**：创建、维护、重置、持久化会话状态
- **消息路由**：将入站消息路由到正确的 Agent 会话中
- **事件分发**：向所有连接的客户端广播 Agent 事件、状态变更
- **Cron 调度**：管理定时任务
- **配置管理**：加载、校验和热重载配置

### Pi Agent Runtime（Agent 运行时）

Agent 运行时负责整体的 AI 调用和推理循环。当 Gateway 收到一条需要 AI 处理的消息，会调用 Agent 运行时来处理。

- **Agent 循环**：接收消息 -> 组装上下文 -> 调用 LLM -> 执行工具 -> 流式回复 -> 持久化
- **模型选择和故障转移**：可以支持多个模型，自动切换
- **系统提示词构建**：从工作区文件、技能、配置组装系统提示词
- **工具管理**：注册、调度、执行 Agent 可用工具
- **流式传输**：将 LLM 的输出流式传输到客户端
- **上下文压缩**：当对话过长时自动压缩上下文

### Channels（通道系统）

通道系统负责与外部消息平台对接。每一个通道都是一个适配器，将平台特定的 API 转换为 OpenClaw 内部的消息格式。

- **消息通道**：WebChat、WhatsApp、Telegram、Discord 等
- **拓展通道**：Teams、Google Chat、Feishu 等

### Tools（工具系统）

工具系统为 Agent 提供了与外部交互的能力：

- Built-in tools
- Browser
- Agent coordination
- Skills
- Extensions
- Automation
- Media and devices

### Memory（记忆系统）

记忆让 Agent 能“记住”之前的信息：

- **文件记忆**：基于 Markdown 的文件持久化记忆
- **向量搜索**：使用嵌入向量进行语义相似度搜索
- **混合搜索**：结合 BM25 全文搜索和向量搜索

## 数据流路径

为了更直观理解架构，可以参考下面一条消息的生命周期，假设用户在 Telegram 上发送了一条消息：“帮我查一下明天的天气，并将天气数据存储到本地文件，同时在推特上发一条天气的推文”：

```
1. 用户在 Telegram 上发送消息
   │  "帮我查一下明天的天气，并将天气数据存储到本地文件，同时在推特上发一条天气的推文"
   ▼
2. Telegram 通道收到消息（嵌入在 Gateway 进程内）
   │  polling-session.ts → TelegramPollingSession 长轮询
   │  grammY 库通过 HTTP 长轮询从 Telegram API 获取更新
   │  grammY 中间件链：update ID 追踪 → 串行化 → 日志
   ▼
3. Handler 层处理
   │  src/telegram/bot-handlers.ts → handleInboundMessageLike()
   │  - 去重检查（shouldSkipUpdate，基于 LRU 的 update 去重）
   │  - 权限检查（群组策略 / DM 策略 / sender 白名单）
   │  - 文本分片缓冲（Telegram 超长消息自动拆分的重组）
   │  - 媒体组缓冲（多图消息合并）
   │  - 防抖（快速连发 / 转发 burst 合并）
   │  - 媒体下载（resolveMedia）
   ▼
4. Message Processor 入口
   │  src/telegram/bot-message.ts → createTelegramMessageProcessor()
   │  - 串联上下文构建与消息派发两大步骤
   ▼
5. 上下文构建
   │  src/telegram/bot-message-context.ts → buildTelegramMessageContext()
   │  - 路由解析（conversation-route.ts → 确定目标 Agent + Session Key）
   │  - DM / 群组访问控制
   │  - Body 解析（mention 检测 / 音频 preflight 转写 / 贴纸缓存）
   │  - Payload 组装为标准化 ctxPayload（FinalizedMsgContext）
   │  - 会话记录持久化（recordInboundSession）
   │  - 发送 ACK reaction（👀）
   ▼
6. Dispatch 层
   │  src/telegram/bot-message-dispatch.ts → dispatchTelegramMessage()
   │  → src/auto-reply/dispatch.ts → dispatchReplyFromConfig()
   │  - 创建 ReplyDispatcher（带流式 typing 指示）
   │  - 解析模型 + 思考级别
   │  - 加载技能快照
   ▼
7. Pi Agent 运行时接管
   │  src/agents/pi-embedded-runner/run.ts → runEmbeddedPiAgent()
   │  - 队列序列化（确保每个会话同一时间只有一个运行）
   │  - 解析 Auth Profile（getApiKeyForModel，支持多 profile 轮换）
   │  - 构建系统提示词（system-prompt.ts）
   │  - 加载会话历史（SessionManager）
   │  - 注册工具集（createOpenClawCodingTools → exec / read / write / edit / process / openclaw_* 等）
   │  - 安全策略过滤（profile / group / sandbox / subagent / owner-only 工具策略管道）
   ▼
8. 调用 LLM（如 OpenAI）
   │  pi-agent-core → streamSimple() 发送 HTTP 请求到模型 API
   │  - 流式接收 token
   ▼
9. Agentic 工具循环（pi-coding-agent createAgentSession 驱动）
   │  LLM 返回 → 解析 tool_use blocks → 执行工具 → 结果返回 LLM → 重复直到纯文本回复
   │
   ├─ 循环第 1 轮：LLM 返回 tool_use
   │  │  exec("curl wttr.in/Beijing?format=j1")
   │  │  → exec 工具：安全审批（allowlist/safeBins）→ sandbox/gateway 执行
   │  │  → 天气 JSON 数据作为 tool_result 返回给 LLM
   │  ▼
   ├─ 循环第 2 轮：LLM 看到天气数据，返回 tool_use
   │  │  write(path="~/weather.txt", content="2026-02-26 北京天气：晴 12°C")
   │  │  → write 工具：写入文件
   │  │  → 写入成功作为 tool_result 返回给 LLM
   │  ▼
   ├─ 循环第 3 轮：LLM 返回 tool_use
   │  │  exec("推特 API 调用命令")
   │  │  → exec 工具：安全审批 → 执行
   │  │  → 推文发送成功作为 tool_result 返回给 LLM
   │  ▼
   └─ 循环第 4 轮：LLM 返回纯文本（finish_reason: "stop"）
      │  → 循环结束，生成最终回复
      ▼
10. 流式传输处理
    │  src/agents/pi-embedded-subscribe.ts → subscribeEmbeddedPiSession()
    │  - assistant 文本流 → 分块（EmbeddedBlockChunker）
    │  - tool 事件 → 工具摘要（formatToolAggregate）
    │  - reasoning 流 → 独立 reasoning lane
    │  - 去重 / 压缩重试 / messaging tool 检测
    ▼
11. 回复通过 Telegram 通道发回（进程内函数调用）
    │  src/telegram/bot-message-dispatch.ts → 双 lane 流式架构
    │  - answer lane：主回复内容（TelegramDraftStream 打字机效果）
    │  - reasoning lane：思考过程（可选显示）
    │  - Markdown → Telegram HTML 格式化
    │  - 分块（Telegram 消息长度限制 4096 字符）
    │  - 通过 Telegram Bot API（HTTP）发送回 Telegram 服务器
    ▼
12. 回复后清理
    │  - ACK reaction 移除（👀 → ✅）
    │  - Status reaction 更新（🧠 思考 / 🔧 工具调用 / ✅ 完成 / ❌ 错误）
    │  - Draft stream 清理
    │  - 群聊历史维护
    ▼
13. 用户在 Telegram 上看到回复：
    "已完成！
     明天北京天气：晴，最高 12°C，最低 3°C
     天气数据已保存到 ~/weather.txt
     推文已发送：'明天北京天气晴好，12°C，适合出行！'"
```
