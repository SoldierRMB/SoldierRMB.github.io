## 为什么选择 WebSocket 而不是 REST

REST 是 Web 应用中最常见的 API 设计风格。在 REST 中，由客户端**主动**发起 HTTP 请求，然后服务器返回响应，最后连接关闭。这种“请求 - 响应”模型非常适合简单、无状态、易于缓存和扩展。

但对于 AI 应用来说，REST 有局限性：

1. **无法实现服务器主动推送**  
    在 Agent 运行过程中，会产生一系列中间事件，例如工具调用、模型返回、工具执行结果。这些事件需要**实时**推送给客户端，而 REST 只能通过轮训来获取，会导致效率很低且延迟高。
2. **请求 - 响应模型不匹配 AI 交互模式**  
    在一次 AI 对话中，可能会持续数分钟，在 REST 中，要么维持一个超长的连接等待响应，要么就要将请求进行拆分，会导致复杂度大幅上升。
3. **多路复用**  
    OpenClaw 可以在一个连接中进行多个操作，而 REST 要实现就要每个操作都拆分为独立的 HTTP 接口。

### WebSocket 的优势

WebSocket 的最大特点是全双工，何以在一次 HTTP 升级握手之后，建立持久的 TCP 连接，双方可以随时主动发送消息。

## OpenClaw 的幂等键设计

在 OpenClaw 中，所有涉及**副作用**的操作（发送消息、发起 Agent 对话、创建投票等）都要求客户端提供 `idempotencyKey` 字段。

幂等键工作流程：

```
时序 1：正常请求
Client ─── req(method="send", idempotencyKey="abc123") ──▶ Server
       ◀── res(ok=true) ────────────────────────────────── Server
       （Server 记住 "abc123" 已处理）

时序 2：重复请求（因为 Client 没收到 res，或网络重连后重发）
Client ─── req(method="send", idempotencyKey="abc123") ──▶ Server
       ◀── res(ok=true) ────────────────────────────────── Server
       （Server 发现 "abc123" 已处理，直接返回之前的结果，不重复执行）
```
